## From messages to an API

The Protocol Buffers module gave you the **nouns** — messages, the shape of your data. This module gives you the **verbs**: a `service` is a named group of methods (`rpc`s), and each method is one request message in, one response message out.

That single decision — *methods, not resources* — is what makes the client feel like a local function call. Here everything comes together into an actual API surface.

<Callout title="The mental model">
A message answers "what does the data look like?" A **service** answers "what can I *do*?" Design the service well — clear method names, dedicated request/response messages, honest error codes — and the generated client becomes a joy to use. Design it carelessly and every consumer pays for it forever.
</Callout>
