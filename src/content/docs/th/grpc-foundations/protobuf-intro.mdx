## protobuf คือ contract

**Protocol Buffers** (protobuf) คือภาษาที่คุณใช้อธิบายข้อมูลและ service ของคุณ และเป็น format ที่ข้อมูลนั้นเดินทางไปด้วย มันเล่น 2 บทบาทพร้อมกัน:

- **ภาษา schema** — ไฟล์ `.proto` ที่ประกาศ message และ service ของคุณในแบบที่ภาษาใดก็ compile ได้
- **wire format** — binary encoding ที่กระชับ ซึ่ง message เหล่านั้นใช้ตอนวิ่งข้าม network

ใน gRPC protobuf *คือ* contract ทุกอย่าง — type safety, codegen, ประสิทธิภาพ — ไหลมาจากมัน

## message คือ field ที่มีหมายเลข

นี่คือ message ที่เล็กที่สุดแต่ยังน่าสนใจ:

```proto
syntax = "proto3";

message User {
  int64  id    = 1;
  string name  = 2;
  string email = 3;
}
```

`= 1`, `= 2`, `= 3` คือ **field number** และมันคือแนวคิดที่สำคัญที่สุดใน protobuf บน wire protobuf **ไม่** ส่งชื่อ field มันส่ง *หมายเลข* ของ field กับค่า ดังนั้น `name = "Ada"` จึงกลายเป็นประมาณว่า "field 2, string, Ada" — tag ไม่กี่ byte บวกกับค่า

การตัดสินใจออกแบบอย่างเดียวนี้ให้ผลตามมา 2 อย่าง:

1. **มันกระชับ** ไม่มีชื่อ key ซ้ำ ไม่มีเครื่องหมายคำพูด ไม่มี whitespace — มีแค่ tag กับค่า
2. **มัน evolve ได้** เพราะ *หมายเลข* คือตัวระบุ field คุณจึง rename `name` เป็น `full_name` ใน `.proto` ได้โดยไม่ทำให้ใครพัง — wire เห็นแค่ field `2` มาตลอด นี่คือรากฐานของการทำ schema evolution อย่างปลอดภัย (มีทั้งบทรออยู่ข้างหน้า)

## ทำไมไม่ใช้ JSON ไปเลย?

JSON เยี่ยมมากสำหรับ public API — อ่านออก, ใช้ได้ทุกที่, setup เป็นศูนย์ แต่สำหรับการเรียกกันระหว่าง service ที่ปริมาณสูง มันมี cost จริง ๆ ที่ protobuf เลี่ยงได้:

| | JSON | Protobuf |
|--|------|----------|
| **ขนาด** | ส่ง key เป็น text ทุกครั้ง | tag ที่เป็นหมายเลข, ค่า binary |
| **cost การ parse** | parse text แบบ reflective | decode binary ที่ขับด้วย schema |
| **schema** | มีก็ได้ drift ง่าย | บังคับ และ compiler enforce |
| **type safety** | ตอน runtime ลุ้นเอา | type ที่ generate มา ตอน compile |
| **การ evolve** | อาศัย convention กับความกล้า | field number ทำให้เป็นระบบ |

trade-off คือความตรงไปตรงมาอีกด้าน: protobuf **คนอ่านไม่ออก** บน wire และคุณต้องมี `.proto` เพื่อแปลความหมายของ byte นั่นเป็นดีลที่ดีสำหรับ traffic ภายใน และเป็นดีลที่แย่สำหรับ public API ที่อยาก `curl` ได้

## workflow แบบ contract-first

วินัยที่ protobuf บังคับคือ **เขียน contract ก่อน** ก่อนที่ทีมไหนจะเขียน code ทั้งสองฝั่งตกลง `.proto` กันก่อน: นี่คือ message เหล่านี้, นี่คือ method เหล่านี้ แล้ว codegen ก็สร้าง type ที่ตรงกันขึ้นมาทั้งสองฝั่ง schema จะ drift เงียบ ๆ ไม่ได้ เพราะทั้ง client และ server generate มาจากมัน

<Callout title="สรุปในบรรทัดเดียว">
Protocol Buffers เป็นทั้งภาษา schema และ binary wire format ที่กระชับ ทริกสำคัญของมันคือ **field number**: wire พก tag ที่เป็นหมายเลข ไม่ใช่ชื่อ — ซึ่งทำให้ message เล็ก, decode เร็ว และ rename ปลอดภัย schema แบบ contract-first นี้คือสิ่งที่ทุก feature อื่นของ gRPC สร้างต่อขึ้นมา
</Callout>
