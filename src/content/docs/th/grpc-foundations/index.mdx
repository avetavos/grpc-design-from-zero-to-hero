## แนวคิดในประโยคเดียว

**gRPC ทำให้โปรแกรมหนึ่งเรียก function ในอีกโปรแกรมหนึ่ง — ที่รันอยู่บนอีกเครื่องหนึ่ง — เหมือนเรียก function ในเครื่องตัวเอง** คุณนิยาม function และข้อมูลของมันครั้งเดียวในไฟล์ `.proto` แล้ว gRPC จะ generate code ฝั่ง client และ server ที่ทำให้ network หายไปจากสายตา

นี่คือทั้งหมดที่ gRPC สัญญาไว้: คุณเขียน `stub.GetUser(id)` แล้ว request จะวิ่งข้าม network ไปรันบน server และคืนผลลัพธ์แบบมี type กลับมา — ไม่มี URL, ไม่ต้อง parse JSON, ไม่ต้องเขียน HTTP เอง

## ทำไมต้องเริ่มที่นี่

ทุกหัวข้อที่ตามมา — streaming, deadline, interceptor, load balancing — ล้วนเป็นผลจากการตัดสินใจออกแบบ 2 อย่างที่ gRPC เลือกไว้ตั้งแต่ต้น:

export const foundationDiagram = `flowchart TB
  proto["Protocol Buffers\n(contract-first schema)"] --> gen["Generated stubs\n(typed client + server)"]
  http2["HTTP/2\n(multiplexed, streaming, binary)"] --> transport["หนึ่ง connection,\nหลาย call พร้อมกัน"]
  gen --> grpc["gRPC"]
  transport --> grpc
  grpc --> features["streaming · deadlines ·\ninterceptors · load balancing"]`;

เข้าใจสองอย่างนี้ — **binary contract** และ **multiplexed transport** — แล้วส่วนที่เหลือของ gRPC จะเลิกดูเหมือนเวทมนตร์ และกลายเป็นผลลัพธ์ที่เห็นแล้วรู้เลยว่าทำไมต้องเป็นแบบนั้น

<Callout title="การเปลี่ยนมุมมอง">
REST ทำให้คุณคิดในแง่ **resource และ URL** (`GET /users/42`) ส่วน gRPC ทำให้คุณคิดในแง่ **method และ message** (`GetUser(GetUserRequest)`) ไม่มีอันไหนดีกว่าเสมอไป — แต่โมเดลแบบเรียก method นี่แหละคือเหตุผลที่ gRPC รู้สึกเป็นธรรมชาติสำหรับการเรียกกันระหว่าง service และทำไม tooling ถึง generate อะไรให้คุณได้เยอะขนาดนั้น
</Callout>
